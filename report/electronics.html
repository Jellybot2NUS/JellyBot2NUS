01

Overview

The major contributions of the electronics subsystem this semester are: the pneumatic actuation of the jellyfish ROV's silicone flaps using two DC pump motors and two 3-way solenoid valves, driven by a custom BlueOS Docker extension over I²C; camera system characterisation to quantify optical distortion introduced by the dome port, and a 3D orientation visualiser BlueOS extension providing real-time roll, pitch, and yaw feedback.

02

Hardware

The electronics subsystem of Jellybot is built around the Blue Robotics Navigator Flight Controller HAT, mounted on a Raspberry Pi 4 Model B. Unlike a conventional ROV where the flight controller directly drives motors, the Navigator in this project serves exclusively as a sensor and attitude data source. Motor and solenoid control is handled by a separate custom system, which preserves the Navigator's role purely for telemetry, orientation sensing, and depth measurement.

2.1 Sensors and Actuators

The listed components form the core sensing and control framework of Jellybot. Each sensor contributes a unique data stream:

  • To provide a user control interface [design specification 3b], the Playstation 4 Controller was selected.
  • To measure depth [design specification 4a], the Blue Robotics Bar02 Sensor was selected.
  • To provide orientation [design specification 4b], the Blue Robotics Flight Navigator was selected.
  • To provide real-time visual feedback for monitoring and analysis [design specification 4d], the Blue Robotics Low-Light USB Camera was selected.
  • To coordinate communication between all onboard sensors [design specification 5], the Raspberry Pi 4 was selected.

    Software Stack

    The software stack runs on BlueOS [a Docker-based middleware operating system] with ArduSub 4.5.3 STABLE providing firmware-level sensor fusion. Communication between subsystems uses the MAVLink protocol, translated to HTTP/JSON by MAVLink2REST, enabling browser-based tools and custom extensions for live sensor data.

    The system is accessible from a topside MacBook at 192.168.2.2 via Ethernet. After an ArduPilot restart, manually pressing "Start Autopilot" in BlueOS is required to restore QGroundControl connectivity.

    3.2 Motor Control Extension (blueos-motori2c-extension)

    A custom BlueOS Docker extension drives the pneumatic actuation system. A Flask REST API (main.py) runs inside a Docker container on port 8084, communicating with the PCA9685 over I²C Bus 6 using smbus2. PWM duty cycle maps linearly: 0–100% ≈ 0–12V motor output.

    Key API Endpoints

    EndpointMethodFunction
    /motor/<ch>/speedPOSTSet channel speed (0–100%)
    /motor/statusGETCheck PCA9685 connection status
    Table 3 — Motor Control Extension API Endpoints

    Motor electrical noise intermittently causes OSError 5 on the I²C bus. The API handles this with 3 retry attempts at 10 ms intervals. Both custom extensions auto-start on boot via a systemd service with a 15-second delay to allow BlueOS core services to fully initialise.

    3.3 Camera Capture Extension (blueos-capture-extension)

    Pneumatic Actuation System

    The pneumatic actuation system inflates and vents the silicone flaps by controlling two DC pump motors and two 3-way solenoid valves via the PCA9685 over I²C. Each flap pair has an independent motor (pressurisation) and solenoid (direction control), allowing coordinated or per-flap operation.

    4.2 Wiring & Connections

    I²C Interface

    The Navigator exposes I²C Bus 6 via a JST GH 4-pin connector. The PCA9685 uses standard 2.54 mm headers. A JST GH cable was stripped and crimped to 2.54 mm pin headers to bridge the two.

    Motor Driver Layout

    • Channel A (OUT1/OUT2) → DC Pump Motor
    • Channel B (OUT3/OUT4) → 3-Way Solenoid Valve

    Solenoid Valve Behaviour

    StatePin ConnectionEffect
    PoweredPin 1 ↔ Pin 2Inflate — pump pressurises flap
    UnpoweredPin 1 ↔ Pin 3Vent to atmosphere — flap deflates

    Safety

    Valve release on emergency stop is instantaneous and mechanical — no software delay. The solenoid vents automatically when de-energised.

    4.3 PS4 Controller Interface

    The controller.py script runs on a Mac and sends HTTP requests to the Pi's Flask API. Button mappings were determined empirically.

    ButtonAction
    L1Pulse Flap Pair 1 (0.7 s)
    R1Pulse Flap Pair 2 (0.7 s)
    SquareActivate Pair 1 (1 s)
    TriangleActivate Pair 2 (1 s)
    XActivate both pairs simultaneously (1 s)
    L2Trap air — Pair 1: motor runs 1 s then off, solenoid holds
    R2Trap air — Pair 2: motor runs 1 s then off, solenoid holds
    OEmergency Stop — instantaneous valve release + motor cutoff, resets all locks
    Table 7 — PS4 Controller Button Mappings

    The L2/R2 trap-air sequence engages a one-press lock that holds the solenoid closed. The lock can only be reset by pressing O.

    Camera Gimbal

    The camera is mounted in a Blue Robotics servo gimbal. PWM range: 1000–1900 µs (inverted mapping) — 1000 µs = +45°, 1900 µs = −45°, neutral at 1450 µs. A servo_neutral.py script is saved at /home/pi/ for quick gimbal centring.

    05

    Component Configuration

    5.1 AHRS Orientation & Accelerometer Calibration

    Because Jellybot is a vertical cylinder, the Navigator is mounted upright inside the enclosure — a significant deviation from the default flat/horizontal orientation assumed by ArduPilot.

    The Problem

    ArduPilot's IMU assumes the board is mounted face-up. Mounting it vertically means gravity is sensed on the wrong axis, making all attitude readings incorrect by default.

    ArduPilot provides the AHRS_ORIENTATION parameter to remap IMU axes. After determining the Navigator's installed orientation, this was set to Pitch90 (value 25), correcting for the 90° pitch offset.

    Accelerometer Calibration

    Performed via QGroundControl's Sensors tab (6-position procedure). The GPIO pins on the Navigator were used as a fixed reference point for identifying the "nose" direction during each step.

    Compass Calibration (Pending)

    Requires physically tumbling the ROV through many orientations while connected. The Ethernet tether constrains tumbling range and introduces magnetic interference. Planned workaround: connect via WiFi hotspot from a phone.

    Note

    At ±90° pitch orientations, gimbal lock is expected to affect roll readings. This is a mathematical property of Euler angles, not a calibration error.

    5.2 Bar02 Depth/Pressure Sensor Integration

    The Bar02 connects to one of the two I²C ports on the Navigator via JST GH cable (3.3V, SCL, SDA, GND — no level converter required). The BARO_PROBE_EXT parameter must be enabled to activate the MS5837 driver, followed by a reboot.

    Depth Calculation

    Depth = (Pressure − Atmospheric Pressure) / (Fluid Density × Gravity)

    Fluid density can be adjusted via EK3_DENSITY for fresh vs. salt water.

    Zeroing Behaviour

    At every power-on, ArduPilot records the current atmospheric pressure as the 0 m reference. Powering on indoors introduces an offset equal to the indoor-to-surface pressure difference.

    Correct Calibration Procedure

    Float Jellybot at the water surface, then run Barometer calibration in QGroundControl (Vehicle Setup → Sensors). This re-zeros the depth reference without requiring a reboot.

    Large depth reading jumps at the water surface are caused by the sensor face intermittently breaking the surface. Readings stabilise once submerged to ~20–30 cm. The Navigator's internal MS5611 barometer is sealed inside the enclosure — it is not usable for depth sensing.

    5.3 Leak Sensor Integration

    Two SOS leak sensor probes connect to the dedicated LEAK ports on the Navigator PCB — simple digital GPIO reads (wet/dry logic). Each probe tip contains a blue porous sponge; two electrical contacts are bridged when wet, triggering an ArduSub leak event.

    ArduSub can be configured via QGroundControl's Safety tab to warn or automatically surface on leak detection. With the Navigator's built-in logic set to "Low when dry", no additional configuration is needed. Probes should be placed near cable penetrators, O-ring seals, and at the lowest point of the enclosure.

    5.4 Battery Monitoring

    A Blue Robotics Power Sense Module sits inline on the POWER port (JST GH 6-pin) between the battery and the Navigator, measuring voltage and current continuously.

    QGroundControl — Power Config: Monitor set to "Analog Voltage and Current", capacity 20,000 mAh, 3S, minimum arming voltage 0.0V (ArduPilot not used for motor arming). Live battery voltage and remaining capacity display in Cockpit, calculated by accumulating current draw against the 20,000 mAh capacity.

    Camera System Characterisation

    6.1 Introduction & Motivation

    Jellybot's camera system consists of a Blue Robotics USB Low-Light Camera sealed behind a clear polycarbonate acrylic dome port. While essential for waterproofing, the dome introduces a refractive air–glass–water interface that can distort imagery.

    For a system whose core task is visual identification of coral bleaching — a colour- and texture-sensitive problem — uncharacterised optical distortion poses a direct risk to detection reliability. Three objectives structure this work:

    1. Quantify geometric distortion introduced by the dome port, in air and underwater.
    2. Characterise spatial sharpness degradation across the field of view (centre, edges, corners).
    3. Evaluate auxiliary lighting effects on underwater image quality.

    6.2 Methodology

    Calibration Target

    All geometric tests use a printed and laminated A4 checkerboard — 9 × 6 inner corners, 30 mm squares. A physically printed target is mandatory; digital displays introduce their own deformation. The target occupies 40–60% of the frame so distortion is measured across the full FOV including edges and corners.

    Capture Setup

    Two lamps at 45° angles minimise specular reflections. Tripod feet are marked for reproducibility. Each session captures 20 structured shots (centre, shifted, tilted, corner regions, near/far). Two sessions per scenario: with dome and without.

    Distortion Pipeline (Scenarios 1 & 2)

    Step 1 — Corner Detection

    cv2.findChessboardCorners()   # Coarse detection
cv2.cornerSubPix()            # Sub-pixel refinement

    Minimum 15/20 accepted images enforced per session.

    Step 2 — Camera Calibration

    cv2.calibrateCamera()

    Yields intrinsic matrix and distortion coefficients [k1, k2, p1, p2, k3]. Sign: k1 < 0 → barrel distortion; k1 > 0 → pincushion. Dome effect: Δk1 = k1_dome − k1_nodome.

    Step 3 — Sharpness Mapping

    laplacian_var = cv2.Laplacian(roi, cv2.CV_64F).var()

    Higher variance = sharper. Percentage sharpness reduction computed per zone (centre / edges / corners).

    Step 4 — Chromatic Aberration (Scenario 2 only) — In water, refractive index mismatch at the dome–water interface causes wavelength-dependent refraction, measured as lateral offset between R and B channels along detected edges.

    Lighting Pipeline (Scenario 3)

    Coral reef print replaces the checkerboard. Evaluated using UCIQE (Underwater Colour Image Quality Evaluation — combining chroma, saturation, and contrast), per-channel RGB histograms, noise level, and vignetting.

    07

    3D Orientation Visualiser

    A custom BlueOS extension provides real-time visual representation of Jellybot's orientation — addressing the lack of visual feedback inherent to operating an upright cylinder underwater.

    What It Does

    A 3D model of Jellybot (cylindrical body, dome top, animated tentacles) renders in the browser; its roll, pitch, and yaw mirror the actual vehicle's live orientation.

    How It Works

    • Every 100 ms, the extension polls MAVLink2REST at /mavlink/vehicles/1/components/1/messages/ATTITUDE.
    • Roll, pitch, yaw (radians, NED frame) map to Three.js axes: Pitch → X, −Yaw → Y, Roll → Z.
    • Linear interpolation (lerp) smoothing prevents jittery motion.
    • HUD overlay shows roll/pitch/yaw in degrees with bar indicators.
    • Green ● Live / red ● No Signal pill shows connection status.
    • Manual drag-to-rotate overrides the live feed for inspection.

    Deployment

    Packaged as a Docker image, pushed to Docker Hub, installed via BlueOS Extension Manager (Kraken). Registers as a sidebar service via the register_service API on startup. Served via nginx, which also proxies MAVLink2REST to avoid CORS issues. Delivered as plain HTML (Three.js) for compatibility with phone and iPad access.

    09

    Key Engineering Decisions

    Why not use ArduPilot GPIO or Servo PWM for motor control?

    ArduPilot's "GPIO" channel mode is a software label only — the physical pins remain PCA9685 I²C outputs. Servo PWM (50 Hz, 1000–2000 µs) encodes position, not duty cycle, making it incompatible with H-bridge motor control. Driving the PCA9685 directly via smbus2 was the correct approach.

    Independent H-Bridge Drivers

    The initial design used a single driver with motors and solenoids in parallel per channel. This was revised to two independent drivers to give individual control over each motor and solenoid, enabling the trap-air sequence and precise per-flap operation.

    Lesson Learned — Wiring Verification

    A motor/solenoid channel swap caused partial failures (individual commands failed; "all channels" worked). This highlighted the importance of empirically verifying wiring rather than relying on assumed pin mappings.

    Component Securing

    Tape-based mounting failed under motor and valve vibration. The final securing method uses hose clamps around motor bodies and epoxy putty at PCB corners — no bracket redesigns needed, reliable mechanical retention achieved.
